# AGENT PLAYBOOK FOR KOMUTRACKER-RUST

Purpose: give automated agents a single place to learn how to build, test, lint, and style code in this repo. Keep it in sync with repo conventions.

## Repository Layout
- Root Makefile orchestrates builds, formatting, linting, packaging.
- `kt-app/`: Tauri 2.x Rust desktop shell (main focus here).
- `kt-server-rust/`: Rust server workspace (kt-server, kt-sync, client, datastore, etc.).
- `kt-webui/`: Vue 2.7 web UI submodule (built by Make). Avoid edits unless you intend to change upstream.
- `kt-watcher-afk/`, `kt-watcher-window/`: Python watcher submodules, installed into `venv/`.
- `modules/`: symlinks to built watcher binaries and kt-sync; regenerated by `make modules`.
- No Cursor or Copilot instruction files found; follow this guide instead.

## Toolchains and Versions
- Rust toolchain pinned to 1.88.0 (see `rust-toolchain.toml`); rustfmt + clippy required.
- Node.js v20.19.0 (`.nvmrc`); npm used (package-lock present).
- Tauri CLI 2.2.x via devDependency.
- Python 3 with venv + Poetry/pip for watchers; Swift used on macOS to build aw-watcher-window helper.

## Build and Run
- Install JS deps: `npm ci` (Make target `node_modules`).
- Full dev loop: `make dev` (runs prebuild, then `npm run tauri dev` with dev config, sets `KT_APP_MODE=dev`).
- Full release build: `make build` (prebuild + `npm run tauri build`, `KT_APP_MODE=release`).
- Prebuild steps: `make prebuild` (builds kt-webui, installs watchers, sets up modules, generates app icon).
- Package artifacts: `make package` (platform-aware packaging after build).
- Build kt-sync only: `make install-sync` (cargo build release in kt-sync).
- Run kt-sync daemon: `make run-sync` (uses built binary).
- Run watchers individually (after venv setup): `make run-watcher-afk` or `make run-watcher-window`.
- Clean watcher artifacts: `make clean-watchers` (removes venv/modules, cleans kt-server-rust target via cargo clean).

## Tests
- No top-level Jest/Playwright etc.; Rust tests live inside crates (notably `kt-server-rust/kt-client-rust/tests/test.rs`).
- Run all server workspace tests: `cd kt-server-rust && cargo test`.
- Run a single Rust test: `cd kt-server-rust && cargo test -p kt-client-rust test_full -- --nocapture` (replace filter as needed).
- Run kt-app tests (none present now) the same way: `cd kt-app && cargo test <filter>`.
- For noisy async tests, keep `-- --nocapture` so logs surface.

## Lint and Format
- Rust format: `make format` (cd kt-app && cargo fmt). Run in server workspace with `cd kt-server-rust && cargo fmt` if you touch it.
- Rust lint: `make check` (cd kt-app && cargo check && cargo clippy). For server crates, run `cd kt-server-rust && cargo clippy --all-targets --all-features`.
- Treat clippy warnings as actionable; prefer `-D warnings` when adding CI equivalents.
- No eslint/prettier config in root; respect existing formatting inside submodules if you edit them (kt-webui has its own `.editorconfig`).

## Development Workflow
- Clone with submodules (`git clone --recursive`) or run `git submodule update --init --recursive` after cloning.
- Use `make modules` after watcher / kt-sync changes to refresh symlinks.
- During `tauri dev`, app reads bundled or `KT_WEBUI_DIR` override for assets; ensure that env var points to an existing path before using it.
- Autostart toggles rely on `KT_APP_MODE` and config file; release mode uses port 5600, dev uses 5666.
- Logging verbosity controlled by `KT_TRACE` (trace) or `KT_DEBUG` (debug) env vars.

## Rust Code Style (kt-app and server crates)
- Formatting: standard rustfmt; keep imports grouped std → external crates → internal modules; prefer alphabetical ordering inside groups when practical.
- Module structure: keep `mod` declarations at top; use `pub(crate)` for internals unless public API is required.
- Naming: snake_case for functions/vars, CamelCase for types; keep enums descriptive (e.g., `ModuleMessage`, `ModuleEntry`).
- Avoid `unwrap`; prefer `expect("context")` or proper error propagation. For fatal states, supply actionable messages.
- Error handling: return `Result` where callers can react; use `warn!/error!` with context before bubbling or exiting. For user-facing errors, surface dialogs via `tauri_plugin_dialog`.
- Logging: use `log` macros (trace/debug/info/warn/error); avoid `println!` outside tests. Logging config lives in `logging.rs`; use `get_log_path` helpers and respect rotation (32MB, keep 5).
- Concurrency: use `OnceLock`, `lazy_static`, `Condvar`, and `Mutex` as already established; avoid new global mutable state without synchronization. When waiting on condvars, loop until predicate is met.
- Drop hooks: note `AppHandleWrapper` and `TrayIdWrapper` notify condvars on drop; maintain this pattern when extending.
- File/system interactions: use `std::fs` helpers; handle platform differences with `#[cfg]` blocks (see `manager.rs` and signal handling).
- Tauri commands: annotate with `#[tauri::command]`; keep signatures minimal, prefer returning serializable types.
- Menu/tray: keep IDs stable strings (`open`, `quit`, `config_folder`, `log_folder`); update menu builders inside manager while holding correct locks.
- Config handling: use `write_formatted_config` for persistence to keep TOML pretty; if adding fields, update defaults and write formatting accordingly.
- Networking: call `is_port_available` before binding; panic early with dialog when conflicts occur.
- Shell arguments: parse module args with `shell_words::split` to respect quoted strings.
- Cross-platform: guard OS-specific code with `cfg(unix)/cfg(windows)` and keep Windows process creation flags (`CREATE_NO_WINDOW`) intact.
- Main entry: keep `#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]` in `src/main.rs` to prevent extra console windows on release builds.
- Tests: place under `#[cfg(test)]` modules; use helpers to spin up in-memory datastore as in `kt-client-rust/tests/test.rs`.
- Logging colors/levels configured in `logging.rs`; if adding new targets, extend `level_for` calls instead of sprinkling ad-hoc filters.

## Tauri/Webview Notes
- `tauri.conf.json` and `tauri.dev.conf.json` live under `kt-app/`; pass explicit `--config` via Make targets (already wired).
- Web assets default to bundled `kt-webui/dist`; when overriding with `KT_WEBUI_DIR`, validate path existence or panic with clear message.
- `tauri_plugin_single_instance` writes `single_instance.lock` under runtime dir; `listen_for_lockfile` removes it and shows window—keep lock semantics if you change paths.
- On close events, window hides instead of exiting; preserve `CloseRequested` handler.

## Configuration and Paths
- Config file path per OS (TOML):
  - macOS: `~/Library/Application Support/komutracker/kt-app/config.toml`
  - Linux: `~/.config/komutracker/kt-app/config.toml`
  - Windows: `%APPDATA%\komutracker\kt-app\config.toml`
- Runtime dir stores `single_instance.lock`; log files in platform-specific cache/log dirs via `logging::get_log_path`.
- Default autostart modules: watcher(s) + `kt-sync daemon`; Wayland swaps watchers for `kt-awatcher`.

## Python Watchers
- Create env: `python3 -m venv venv && ./venv/bin/pip install --upgrade pip` (handled by `make venv`).
- Install: `make install-watchers` (installs both) or individual `make install-watcher-afk` / `make install-watcher-window`.
- macOS builds a Swift helper binary (`aw-watcher-window-macos`); only built on Darwin.
- Watchers run via venv entrypoints; modules symlinked into `modules/` for discovery by manager.

## Submodules and Assets
- Treat submodules (`kt-webui`, `kt-server-rust`, watchers) as upstream-controlled; avoid committing generated files inside unless explicitly updating them.
- When updating submodules, run `git submodule update --init --recursive` and commit the pointer change only.
- Icon generation depends on `kt-webui/media/logo/logo.png`; `make prebuild` creates `kt-app/icons/icon.png`.

## Git and Review Hygiene
- Do not amend or force-push unless instructed. Keep user changes intact.
- Prefer small, focused commits with passing lint/tests. Stage generated files only when necessary.
- If adding secrets/config, never commit `.env` or credentials; config paths listed above live in user directories, not repo.

- `make dev` – run app in dev with hot reload.
- `make build` – release build with bundled assets.
- `make check` – cargo check + clippy for kt-app.
- `make format` – rustfmt kt-app.
- `cd kt-server-rust && cargo test` – server workspace tests.
- `cd kt-server-rust && cargo test -p kt-client-rust test_full -- --nocapture` – run single test.

## kt-app Architecture Cues
- App entry: `src/main.rs` only calls `kt_app_lib::run`; keep `windows_subsystem` attribute intact.
- `lib.rs` holds initialization, config IO, lockfile watching, and tray setup; follow patterns when adding new app-level state.
- `manager.rs` manages module processes; keep menu IDs stable and use condition variables before touching the app handle.
- Config defaults live in `UserConfig::default`; update discovery paths + autostart lists there when adding modules.
- Notifications and dialogs go through Tauri plugins; avoid raw OS calls.
- Avoid busy loops: prefer existing watcher pattern (`try_recv` + short sleep) when polling.
- When adding commands exposed to JS, register via `tauri::generate_handler!` and ensure return types are serializable.
- Keep runtime paths and config paths consistent with `dirs.rs`; add helpers there if new paths are required.

## kt-server-rust Workspace Notes
- Workspace root: `kt-server-rust/` with crates `kt-server`, `kt-client-rust`, `kt-datastore`, `kt-sync`, `kt-query`, `kt-transform`, `kt-models`.
- Build full workspace: `cd kt-server-rust && cargo build --release` (tests use port 5666 by default for dev).
- Run server binary directly: `cd kt-server-rust && cargo run --bin kt-server` (testing mode on port 5666).
- `kt-sync` lives under `kt-server-rust/kt-sync`; build release with `cd kt-server-rust/kt-sync && cargo build --release`.
- Server tests rely on Rocket; keep ports configurable and avoid hard-coding 5600 where possible.
- Data access usually goes through `kt-datastore::Datastore`; prefer in-memory datastore for tests.

## Frontend/Web Assets
- `kt-webui` is a Vue 2.7 submodule; built via `make prebuild`/`make build`. Avoid editing unless updating upstream.
- Assets for Tauri come from `kt-webui/dist` or `KT_WEBUI_DIR`; validate the path before passing to Tauri builder.
- Icons generated via `npm run tauri icon ./kt-webui/media/logo/logo.png`; keep source asset intact.

## Logging and Telemetry Details
- Logging output formatted in `logging.rs` with colored levels; levels controlled by `KT_TRACE`/`KT_DEBUG` env vars.
- Log rotation: 32MB max with 5 retained rotations; reuse `rotate_log_if_needed` if adding new log files.
- Prefer `info!` for lifecycle, `trace!` for frequent events, `warn!` for recoverable issues, `error!` for failures.
- Avoid mixing stdout/stderr prints with fern logger; use `eprintln!` only before logging is initialized.

## Error Handling Patterns
- Bubble errors with context; use `expect("Failed to …")` where failure is unrecoverable.
- For user-visible issues (ports in use, malformed config), show dialogs via `tauri_plugin_dialog` and then bail fast.
- When performing IO (config/log paths), create parent dirs via `create_dir_all` before writes; propagate errors upward.
- When reading env vars, handle `Result`/`Option` instead of panicking; only panic when continuing would corrupt state.

## Concurrency and Process Management
- Globals use `OnceLock` + `lazy_static` + `Condvar`; notify condvars in `Drop` implementations to wake waiters.
- Guard shared state with `Mutex`; log lock errors via `expect` with meaningful text to ease debugging.
- Module processes start via `Command` and are tracked in `ManagerState`; update maps consistently when adding new module types.
- Respect OS-specific termination paths: `send_sigterm` uses nix on Unix and WinAPI on Windows; extend with matching cfg guards.
- Restart logic caps at 3 retries; keep counter/reset logic aligned with dialog notifications.

## Configuration File Expectations
- Config is TOML written via `write_formatted_config`; preserve pretty printing (modules on separate lines, discovery paths array formatting).
- Default ports: dev 5666, release 5600; keep in sync with server config when making changes.
- Autostart modules list differs on Wayland (uses `kt-awatcher`); check `XDG_SESSION_TYPE`/`WAYLAND_DISPLAY` when modifying.

## Dependency Management
- Rust dependencies declared per crate; avoid workspace-wide `cargo update` unless coordinated.
- JS deps managed via npm + package-lock; use `npm ci` for reproducibility.
- Python deps installed via `pip install .` inside watcher submodules; keep `venv/.afk-installed` and `.window-installed` markers intact.

## Troubleshooting Tips
- Port conflicts: `is_port_available` checks 127.0.0.1; show dialog and exit if port is taken.
- Tray/menu issues: ensure `TrayId` is initialized and condvars signaled before menu mutation.
- Lockfile handling: `listen_for_lockfile` watches runtime dir for `single_instance.lock`; keep watcher polling interval reasonable.
- On macOS, app uses `ActivationPolicy::Accessory`; avoid reverting unless you need a dock icon.
- Windows builds hide console via `CREATE_NO_WINDOW`; maintain when adjusting process spawning.

Stay consistent with existing patterns; prefer extending helpers (dirs/logging/manager) over reimplementing behavior elsewhere. When in doubt, mirror existing log/error wording and keep platform-safe guards.
